# PalmPalm 引き継ぎドキュメント (2026-03-12 更新版)

## 0. 目的

このドキュメントは、次の担当者が最短で開発再開できるように、
「現状の実装」「確認済み挙動」「未解決課題」「次アクション」をまとめたものです。

---

## 1. プロジェクト概要

AI手相占い「ぱむぱむ」のハッカソン向けアプリ。
ユーザーの身体反応（振動センサー）を `agitation_level` として数値化し、
AIの占いリアクションに反映して盛り上げる構成。

### コアコンセプト

- センサー反応 = 占いが当たっている証拠
- `level` が高いほどAIが強く追い込む
- 急上昇 (`is_spike`) 時は割り込みリアクション

---

## 2. システム構成（実装ベース）

```text
Raspberry Pi / Pico (振動パルス "1")
    ↓ WebSocket /ws/sensor
FastAPI Backend (:8000)
    ├─ AgitationEngine
    ├─ SessionManager
    │   ├─ MockGeminiSessionManager (MOCK_MODE=true)
    │   └─ TwoStageSessionManager   (MOCK_MODE=false)
    ├─ WebSocket /ws/frontend
    ├─ POST /api/audio
    └─ StaticFiles /audio/*
         ↓
React + Vite Frontend (:5173)
    ├─ useBackendWS (agitation_update / ai_text / ai_audio)
    ├─ useVAD (MicVAD + float32ToWav + POST /api/audio)
    └─ 4画面 (Title -> Rules -> Session -> End)
```

### モード差分

| モード | 有効クラス | AI出力 | 備考 |
|---|---|---|---|
| `MOCK_MODE=true` (docker-compose デフォルト) | `MockGeminiSessionManager` | `ai_text` + `ai_audio` | 台本再生 + ランダム振動 |
| `MOCK_MODE=false` | `TwoStageSessionManager` | `ai_text` のみ | Gemini通常APIで2段階生成 |

---

## 3. 主要データフロー

### 3.1 センサー入力

1. `/ws/sensor` で `"1"` を受信
2. `AgitationEngine.record_pulse()`
3. `snapshot()` で `level/trend` 算出
4. `agitation_update` を `/ws/frontend` に配信
5. `engine.is_spike()` が真なら `send_push(level, trend)` を非同期実行

### 3.2 マイク入力

1. `SessionPage` で `useVAD({ httpBase })` 実行
2. 発話終了時 (`onSpeechEnd`) に `Float32Array -> WAV` 変換
3. `POST /api/audio` にバイナリ送信
4. Backend が `gemini.receive_audio(audio_bytes, "audio/wav")` を非同期実行
5. `ai_text`（モック時は加えて `ai_audio`）をWS配信

### 3.3 フロント状態管理

- `useBackendWS` が以下を集約
  - `agitationLevel`
  - `agitationTrend`
  - `aiText`（チャンクを追記）
  - `aiAudioUrl`
  - `connected`
- `SessionPage` は `aiAudioUrl` 更新で `new Audio(url).play()` を実行

## 5. 未解決課題・技術的注意点

### 5.1 VAD 初期化エラー（最優先）

症状:

- `useVAD` 内 `MicVAD.new()` で初期化失敗するケースがある
- UI上は `VAD: <error>` として表示される

既に実施済み:

- `frontend/public` に WASM/ONNX/Worklet を配置
- `vite.config.js` の `optimizeDeps.exclude = ['onnxruntime-web']`
- `@ricky0123/vad-web` を動的 import に変更

次の確認ポイント:

- ブラウザコンソールの `[VAD] init error:` の実メッセージ
- `workersOptions.url` と `modelURL` の実際の fetch 成否
- 必要なら `@ricky0123/vad-web` も `optimizeDeps.exclude` に追加

### 5.2 本番モードで `ai_audio` が出ない

- `TwoStageSessionManager` は現状 `ai_text` のみ配信
- `SessionPage` の音声再生は、実質モックでのみ機能
- 本番で喋らせるなら以下のどちらかが必要
  - TTS をバックエンド追加して `ai_audio` を配信
  - フロントでテキスト読み上げ（Web Speech API 等）

### 5.3 セッション再入場時の表示リセット不足

- `aiText` が `useBackendWS` で追記のみ
- `End -> Title -> Session` で前回テキストが残る可能性あり
- 画面遷移時に `aiText` を明示リセットする設計が必要

### 5.4 Pico GPIO ピン記述の不整合

- `pico/main.py` の実装: `GPIO_PIN = 16`
- しかし同ファイルコメントと `pico/README.md` は GP17 と記載
- 配線ミス防止のため要統一

### 5.5 Lint 設定未整備

- `frontend/eslint.config.js` が JSX パース設定不足
- ビルドは通るが静的解析が実運用不能

### 5.6 `backend/src/main.py` の `/audio` 相対パス依存

- `app.mount("/audio", StaticFiles(directory="assets/audio"), ...)`
- 実行カレントディレクトリが `backend/` 以外だと参照失敗しうる
- `Path(__file__)` 起点へ変更すると安全

## 6. 再開時の最短コマンド

### 6.1 まず動かす（モック）

```bash
docker compose up --build
```

- Frontend: `http://localhost:5173`
- Backend health: `http://localhost:8000/health`

### 6.2 バックエンドテスト

```bash
cd backend
uv run pytest tests -v
```

### 6.3 実機センサー系（必要時）

- `MOCK_MODE=false` で backend 起動
- Pico 利用時は `pico/serial_bridge.py` で `/ws/sensor` へ転送
- Raspberry Pi 利用時は `raspi/sensor.py` を backend 向け起動

## 8. 重要ファイルマップ

### Backend

- `backend/src/main.py`
  - FastAPIエントリ、WS/HTTPルーティング、SessionManager切替
- `backend/src/agitation_engine.py`
  - 動揺率・トレンド・スパイク判定ロジック
- `backend/src/mock_gemini_session.py`
  - モック台本、`ai_text/ai_audio` 配信
- `backend/src/two_stage_session.py`
  - Gemini通常APIで2段階テキスト生成
- `backend/tests/*.py`
  - 上記コアロジックの単体テスト

### Frontend

- `frontend/src/hooks/useBackendWS.js`
  - WS受信と状態集約
- `frontend/src/hooks/useVAD.js`
  - マイク検知と `/api/audio` POST
- `frontend/src/pages/SessionPage.jsx`
  - セッション中UI、音声再生、VADエラー表示
- `frontend/src/components/VibrationEffect.jsx`
  - 動揺率連動エフェクト（初心者編集ポイント）

### Device

- `pico/main.py`
  - Pico MicroPython: 振動検知でシリアル `1` 出力
- `pico/serial_bridge.py`
  - Mac上ブリッジ: シリアル -> `/ws/sensor`
- `raspi/sensor.py`
  - Raspberry Pi GPIO -> `/ws/sensor`（非RPi環境ではモック送信）
